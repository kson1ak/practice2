## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C 
```

**Объяснение:**

| Шаг | Что выполняется | Вывод | Объяснение |
|-----|----------------|-------|------------|
| 1 | Статический блок `static {}` | `A` | Выполняется **один раз** при загрузке класса в память |
| 2 | `System.out.print("D ")` в `main()` | `D` | Первая инструкция метода `main` |
| 3 | Блок инициализации `{}` (перед конструктором) | `B` | Выполняется **перед каждым** вызовом конструктора |
| 4 | Конструктор `Demo()` | `C` | Выполняется после блока инициализации |
| 5 | `System.out.print("E ")` в `main()` | `E` | После первого создания объекта |
| 6 | Блок инициализации `{}` (снова) | `B` | Перед вторым вызовом конструктора |
| 7 | Конструктор `Demo()` (снова) | `C` | Второй объект |

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**

В Child (наследник из другого пакета) доступны: public члены, protected члены.
НЕ доступны: private члены, package-private члены (без модификатора).

В другом классе пакета b, который НЕ наследует Parent, доступны: только public члены.
НЕ доступны: private члены, package-private члены, protected члены

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**

String Pool (пул строк) — это специальная область памяти в куче (Heap), где JVM хранит уникальные строковые литералы. Если строка уже есть в пуле, новая не создаётся — возвращается ссылка на существующую.

Метод intern() возвращает каноническую ссылку на строку из пула:
Если строки нет в пуле → добавляет её туда и возвращает ссылку. Если строка уже есть в пуле → возвращает ссылку на строку из пула

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:**

Проблема: Класс C наследует два одинаковых метода по умолчанию (default) от интерфейсов A и B. Возникает конфликт имён (diamond problem). Компилятор не знает, какой метод hello() использовать.

Как исправить:

Способ 1: Переопределить метод в классе C
```
class C implements A, B {
    @Override
    public void hello() {
        System.out.println("C");  // своя реализация
    }
}
```
Способ 2: Вызвать конкретную реализацию через `super`
```
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // вызываем метод из A
        // или B.super.hello();
    }
}
```
Способ 3: Сделать один из интерфейсов не default (если возможно)
```
interface A { void hello(); }  // абстрактный метод
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B {
    @Override
    public void hello() {
        B.super.hello();  // теперь понятно
    }
}
```
Способ 4: Использовать один интерфейс вместо двух
```
interface A extends B { }  // A наследует B
class C implements A { }   // конфликта нет
```

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**

Главное отличие:

StringBuffer — все методы синхронизированы (synchronized), поэтому потокобезопасен, но медленнее

StringBuilder — не синхронизирован, поэтому быстрее, но не подходит для многопоточности

int x = 10;

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**

`EnumMap` быстрее `HashMap`, потому что внутри он хранит данные в обычном массиве (индексируется по `ordinal()` enum-ключа), а не в хэш-таблице с бакетами, что даёт O(1) без вычисления хэша и обработки коллизий.

Внутренняя реализация (одно предложение):

`EnumMap` хранит значения в простом массиве `Object[] vals`, где индекс элемента соответствует порядковому номеру (`ordinal()`) enum-константы, что позволяет получать и вставлять элементы по прямому индексу без вычисления хэш-кода и разрешения коллизий.

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**

Для `record Point(int x, int y) {}` компилятор автоматически создаёт:

1) Конструктор с параметрами `(int x, int y)`
2) Геттеры — `x()` и `y()` (не `getX()`!)
3) `equals()` — сравнивает все поля
4) `hashCode()` — учитывает все поля
5) `toString()` — формат Point[x=10, y=20]

Добавить в record изменяемое поле нельзя. Все поля в `record` — финальные (`final`) по определению. Record предназначен для неизменяемых (immutable) данных.

```
record Point(int x, int y) {
    int z;  // Ошибка! Все поля должны быть в заголовке
}
```

Наследоваться от record нельзя. `record` — это неявно `final`. Его нельзя расширить:
```
record Point(int x, int y) { }
record Point3D extends Point { }  // Ошибка! Cannot inherit from record
```
